Mode: $ARGUMENTS

Starts with `--pre` → Pre-creation mode. Contains `--fix` → Fix mode (audit + apply). `--drift` → Drift Check. `--orphans` → Orphan Check. Otherwise → On-demand audit (read-only).

## On-demand Audit Mode (`/md-check`)

### Step 1 — Inventory
```bash
{ find "$HOME/.claude" -maxdepth 1 -name "*.md"; find "$HOME/.claude/docs" -maxdepth 1 -name "*.md" 2>/dev/null; } | sort | xargs wc -l 2>/dev/null
```
Flag any file over 200 lines as `OVERSIZE`.

### Step 2 — Header Map
Extract section headers per file:
```bash
grep -nE "^#{1,3} " <file>
```

### Step 3 — Topic Overlap Detection
Per file, extract top-20 frequent non-stopword tokens (4+ chars):
```bash
grep -oE '\b\w{4,}\b' <file> | tr '[:upper:]' '[:lower:]' | sort | uniq -c | sort -rn | head -20
```
Per file pair: overlap % = |intersection of top-20 term sets| / |smaller set|.
- `>40%` overlap → flag as merge candidate
- Shared section header (exact match) → counts double, flag immediately

Exclude: `node_modules/`, `plugins/`, `brainstorms/`

### Step 4 — Duplicate Rule Detection
Normalize each bullet/numbered line (lowercase, strip markdown punctuation). Find near-identical lines (>80% token match) in 2+ files. Report as: `"<rule text>" → found in <file1>:line + <file2>:line`

### Step 5 — Output Tables

**File inventory** — Status vocab: `OK` / `OVERSIZE` (>200 lines) / `DUP-RULE` / `OVERLAP`:
| File | Lines | Status | Note |
|------|-------|--------|------|
| CODE_REVIEW.md | 25 | DUP-RULE | → CORE_RULES.md:3 |

**Merge candidates:**
| File A | File B | Overlap % | Shared headers |
|--------|--------|-----------|----------------|

**Duplicate rules:**
| Rule text (truncated) | File A:line | File B:line |
|-----------------------|-------------|-------------|

No merge candidates or duplicate rules → print: `✓ No overlaps or duplicates found.`

## Drift Check Mode (`/md-check --drift`)

A skill's `description:` in its SKILL.md is the TRUTH. Five hand-maintained catalog rows describe the same skill (`STORIES`, `TAGLINES`, `TAGLINES_SHORT`, `WHEN_TO_USE`, `WHY_TO_USE`) — edit one side and the other silently lies. `RENDERED*.md` are generated by `regen.py`: never check or edit them; fix the sources and regen. `docs/SKILL_TRIGGERS.md` holds no per-skill rows — routing rule only.

**Do NOT re-read every catalog file and eyeball them** — one agent judging the whole library samples instead of enumerating (two runs once returned disjoint findings). Both steps below exist to kill that failure.

### Step 1 — Mechanical: what CHANGED (deterministic, free, complete)

```bash
python3 ~/.claude/skills/my-skills/catalog_lock.py --check
```

`catalog-lock.json` holds a sha256 of every description and row as of last verification. Exit 0 = nothing moved: **stop, report `✓ catalog lock current`, do no LLM work.** Exit 1 prints exactly what moved:

| Report | Means |
|---|---|
| `TRUTH CHANGED: <skill>` | SKILL.md description edited — its rows may now lie |
| `ROW CHANGED: <skill> @ <file>` | a catalog row was edited — may now misdescribe the skill |
| `STRUCTURE CHANGED: <skill>` | SKILL.md `## ` headings moved — its `rel:` row describes the pipeline, so it may now lie |
| `UNLOCKED: <skill>` | never verified — needs a Step 2 pass |
| `ROW ADDED / REMOVED` | a catalog gained or lost a row |

`rel:` rows describe INTERNALS, so their truth is the SKILL.md body — but only the `## ` heading skeleton is hashed (the pipeline IS the headings): heading moves flag, prose typos stay silent.

### Step 2 — Judgment: ONLY on what Step 1 flagged

For each flagged skill, compare description against rows and judge:
- `OK` — accurate, even if terser
- `DRIFT` — omits a key behavior, or describes a removed one
- `WRONG` — states something false (a flag/command that does not exist, or the opposite)

**Bound the work**: one skill at a time, or subagents at **≤7 skills each** — never one agent over the whole library; an unbounded pass silently samples.

### Step 3 — Fix, regen, re-lock

Fix the rows (never `RENDERED*.md`), then:
```bash
python3 ~/.claude/skills/my-skills/regen.py
python3 ~/.claude/skills/my-skills/catalog_lock.py --relock
```
Re-locking makes the next run's green meaningful; `verify.sh` fails on a stale lock, so an unlocked change cannot ship.

### Output
```
| Skill | Source file | Row says | SKILL.md says | Verdict |
```
Only non-OK rows. Then: `X skills flagged by lock, Y drifted, lock re-sealed`.

### What this does NOT catch — state it in the output

- **A row that was always wrong and never edited.** The lock proves *nothing moved*, not *everything is true* — only a full baseline sweep (Step 2 over every skill) establishes truth.
- **A `rel:` row that goes stale without the headings moving** — the deliberate price of not flagging on every typo.
- **`bolt:` rows** (`fallow`, `gh`, `draw`) — external tools, no SKILL.md to hash against. Three rows, rarely change; check by hand.
- **`rel:` rows for untracked third-party skills** (improve, trim*) — no tracked SKILL.md to compare against.
- Never report "no drift" on a green lock. Report "nothing changed since last verified."

## Orphan Check Mode (`/md-check --orphans`)

Two-way check: dangling references to skills that no longer exist, and real skills nobody can discover. This should have caught the impeccable situation before it shipped.

### Step 1 — Real skills on disk
```bash
ls -d ~/.claude/skills/*/ | xargs -n1 basename
```
Ground truth: a skill is "real" only if listed here AND has a readable `SKILL.md` (plugin-managed dirs with a live SKILL.md count — the check is existence, not authorship).

### Step 2 — Names mentioned in catalogs/docs
```bash
grep -hoE "^[a-z][a-z0-9-]+\|" ~/.claude/skills/my-skills/{TAGLINES,STORIES,WHEN_TO_USE,WHY_TO_USE,CATEGORIES}.md 2>/dev/null | tr -d '|' | sort -u
```
Also scan prose mentions of `/name` in README.md and other skills' SKILL.md bodies (`grep -oE '/[a-z][a-z0-9-]+' <file>`) — a skill can be dangling-referenced in plain text without a catalog row.

### Step 3 — Dangling references (Step 2 names not in Step 1)
Per name mentioned but absent on disk: `⚠️ DANGLING — "<name>" referenced in <file>:line but no skills/<name>/SKILL.md exists` (the `/impeccable` failure: describing removed functionality as if it exists).

### Step 4 — Unreferenced skills (Step 1 names not in Step 2, and not in SKILL_TRIGGERS.md)
Per real skill with zero catalog/trigger mentions: `⚠️ UNREFERENCED — "<name>" exists at skills/<name>/SKILL.md but isn't surfaced anywhere a user would find it`. Skip skills with `user-invocable: false` — intentionally internal.

### Step 5 — Output
```
| Verdict | Name | Where |
|---|---|---|
```
Only DANGLING and UNREFERENCED rows. If both empty: print `✓ No orphans — everything on GitHub is real, everything real is findable.`

## Pre-creation Mode (`/md-check --pre <filename>`)

Called by other skills before writing a new MD file — checks if the proposed topic already exists. Input: `--pre <proposed-filename>` (e.g. `--pre ANTISLOP.md`)

1. Run the header map (Step 2 above) across all existing `~/.claude/*.md`
2. Compare the filename stem (e.g. `antislop`) against existing headers and filenames (case-insensitive, fuzzy: strip hyphens/underscores)
3. If proposed content piped via stdin: compute term overlap against existing files

Output — exactly one verdict line, then one line of reasoning (e.g. `MERGE-INTO-EXISTING: ~/.claude/docs/UI_MOCKUPS.md — "antislop" maps to existing "Slop fingerprint" section (header match)`):
- `PROCEED` — no significant overlap found
- `MERGE-INTO-EXISTING: <path> — <reason>` — topic already covered, add to this file instead
- `DUPLICATE-DETECTED: <path>#<section> — <reason>` — content duplicates an existing section

## Notes
- Exclude `brainstorms/` and `plugins/` from all scans — ephemeral/external
- All paths absolute in output
- Restart session required to apply changes to CLAUDE.md or other loaded MDs (no hot-reload)

## FIX MODE (`--fix`)

Default `/md-check` REPORTS; `--fix` reports THEN applies. Audit → fix table → one approve-all gate → apply → re-verify.

### Step 1 — Audit
Run the On-demand audit AND Drift Check. Collect: OVERSIZE files, merge candidates, duplicate rules, drifted descriptions. If fully clean → print `✓ Nothing to fix — all MD files healthy.` and STOP.

### Step 2 — Fix plan
One row per finding:
```
| # | Target | Problem | Fix | Kind |
```
Kind vocab:
- **DEDUPE** — delete the duplicate line(s); keep the canonical copy (name which file:line stays)
- **MERGE** — fold the smaller file's unique content into the larger, delete the smaller (only when overlap >40%)
- **TRIM** — an OVERSIZE file: propose what moves out and where. Append-only logs (`DAILY_CHANGELOG.md`, `learnings.md` §6) are EXEMPT — never trim them.
- **DRIFT-FIX** — rewrite the CLAUDE.md description line to match its SKILL.md `description`

### Step 3 — Approve-all gate
Skip if `$ARGUMENTS` contains `--auto`. Print the fix table, then: `Reply "go" to apply all, or list the # to skip.` Wait for the reply. This mutation gate is the only pause.

### Step 4 — Apply
Apply each approved fix with Edit/Write.
**Personal-file guard (HARD — never edit):** `memory/` `brainstorms/` `plans/` `proposals/` `projects/` `sessions/` `settings.json` `settings.local.json`. Skip + note any fix that would touch these. Never delete a file you did not read this run.

### Step 5 — Verify + report
Re-run the audit. Report before/after:
```
| Metric | Before | After |
| Files >200 lines | | |
| Duplicate rules | | |
| Drifted descriptions | | |
```
Then a table of what changed (file, action). End `✅ Fixed N findings across M files`.
> Edits to `CLAUDE.md` or other loaded MDs need a session restart to take effect (no hot-reload). Note this if any were touched.
